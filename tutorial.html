01 — Por que Minion Engine?

Seu Time de AI Rodando Localmente

Como o Stripe Minions, mas rodando na sua maquina com Docker sandboxes em vez de AWS. Cada acao roda isolada em um container — seu codigo esta seguro.

🐳

Sandbox-First

Cada step roda isolado em Docker. Se algo der errado, o container e destruido. Seu codigo original permanece intocado.

🤖

AI-Powered

Claude Code planeja, implementa, revisa e corrige — autonomamente. Voce define o workflow, os minions executam.

🔄

Self-Healing

Lint falhou? Testes quebraram? Os minions auto-corrigem e tentam de novo. Gates com retry automatico ate 3 rounds.

02 — Conceito

O que sao Minions?

Em vez de um unico agente de AI com acesso total ao seu sistema, voce tem multiplos minions especializados que trabalham em sandboxes isolados. Se um minion cometer um erro, o container e destruido — impacto zero.

Filosofia: Engine controla, Minions executam
Engine: "Minion, implemente o plano" Minion: *implementa dentro do container* Engine: "Obrigado. Agora EU rodo lint." ← deterministico Engine: "Lint falhou. Minion, corrija." Minion: *corrige dentro do container* Engine: "Lint passou. EU rodo testes." Engine: "Tudo OK. Copiando resultados de volta." Engine: "Container destruido. Seu projeto esta seguro."

Por que isso importa?

AI agents tradicionais tem acesso direto ao seu filesystem. Se o agente deletar algo errado, instalar um pacote malicioso, ou corromper sua build — voce limpa a bagunca manualmente. Com sandbox, o pior que acontece e voce destruir o container e tentar de novo.

03 — Docker Sandbox

🐳 Como Funciona o Sandbox

Tres modos de sandboxing. Escolha o nivel de isolamento ideal para seu caso.

🐳

Modo 1: Default (FullWorkflow)

FullWorkflow — tudo roda dentro do Docker por padrao. O workspace inteiro e copiado para o container. Resultados copiados de volta ao final.

minion execute workflow.yaml
# sandbox ON por padrao!

# para desativar:
minion execute workflow.yaml --no-sandbox
🤖

Modo 2: AgentOnly

config.agent.sandbox: true — somente steps de AI (agent) rodam isolados. Commands rodam localmente.

config:
  agent:
    sandbox: true
⚙

Modo 3: Devbox

Full config — imagem customizada, regras de rede, limites de recursos. Controle total.

sandbox:
  enabled: true
  image: "node:20"

Configuracao Completa do Devbox

config:
  global:
    sandbox:
      enabled: true
      image: "minion-sandbox:latest"              # imagem com git,gh,node,cargo,claude

      # Credenciais: env vars do host forwardados ao container
      # Se omitido, auto-forward: ANTHROPIC_API_KEY, OPENAI_API_KEY, GH_TOKEN, GITHUB_TOKEN
      env:
        - ANTHROPIC_API_KEY
        - GH_TOKEN
        - NPM_TOKEN

      # Volumes montados read-only (credenciais do host)
      # Se omitido, auto-monta: ~/.config/gh, ~/.claude, ~/.ssh, ~/.gitconfig
      volumes:
        - "~/.config/gh:/root/.config/gh:ro"
        - "~/.claude:/root/.claude:ro"
        - "~/.ssh:/root/.ssh:ro"
        - "~/.gitconfig:/root/.gitconfig:ro"

      # Excluir do workspace copy (economiza tempo)
      exclude:
        - node_modules
        - target
        - .git/objects

      dns: ["8.8.8.8"]                            # DNS confiavel

      network:
        allow: ["api.anthropic.com", "api.github.com"]  # whitelist
        deny:  ["0.0.0.0/0"]                       # bloqueia tudo mais
      resources:
        cpus: 2.0
        memory: "4g"
🐳 Zero-Config Smart Defaults

Se voce nao especificar env: ou volumes:, o engine faz tudo automaticamente: forwarda ANTHROPIC_API_KEY, GH_TOKEN, monta ~/.config/gh, ~/.ssh e ~/.gitconfig (read-only) e ~/.claude (read-write, para sessoes do Claude CLI). O gh, claude, git push e API calls funcionam out-of-the-box.

Imagem de Referencia — Auto-Build

Zero configuracao de Docker

Na primeira execucao, se a imagem minion-sandbox:latest nao existir, o engine constroi automaticamente. O Dockerfile esta embutido no binario — funciona mesmo instalando via cargo install. Voce nao precisa clonar o repositorio nem rodar docker build manualmente.

A imagem inclui: git, gh, jq, curl, node 20, npm, cargo, clippy, python3, pip, claude CLI.

Se preferir construir manualmente (opcional):

docker build -f Dockerfile.sandbox -t minion-sandbox:latest .

05 — Cenario 1

Entrei num projeto — quero analisar uma PR

Voce acabou de entrar no time e quer revisar a PR #142. O minion analisa arquivo por arquivo, em paralelo, dentro de um sandbox.

Terminal — ~/projects/my-saas
$ cd ~/projects/my-saas $ minion execute code-review --verbose -- 142 🐳 Sandbox created (node:20, 2 CPUs, 4GB RAM) workspace copied to /workspace ✓ fetch_pr_diff 1.4s cmd gh pr diff 142 ✓ list_changed_files 0.3s cmd ◉ review_files 28.7s map(4) 8 files, concurrency:4 ✓ src/api/auth.ts 6.2s agent 2 critical ✓ src/api/users.ts 5.8s agent 1 medium ✓ src/lib/db.ts 7.1s agent clean ✓ src/lib/cache.ts 4.3s agent 1 low ✓ ... +4 files ● consolidate_report 8.3s chat Critical → High → Medium → Low ✓ post_pr_comment 1.1s cmd gh pr comment 142 ✓ Complete in 40.1s — 12.3k tokens — $0.06 🐳 Results copied back. Container destroyed.

O que aconteceu passo a passo

1

🐳 Container Docker criado, workspace copiado

Todo o diretorio do projeto e copiado para /workspace dentro do container.

2

gh pr diff 142 → busca o diff da PR

Roda dentro do container. O diff completo e salvo no Context Store.

3

Lista arquivos alterados

Extrai a lista de arquivos do diff para iterar depois.

4

Para CADA arquivo (paralelo, concurrency:4), AI revisa

Bugs, seguranca, performance, boas praticas. Cada review roda em paralelo com ate 4 minions simultaneos.

5

Consolida em relatorio (Critical → High → Medium → Low)

Um step chat (barato e rapido) sintetiza todas as reviews em um relatorio ordenado por severidade.

6

Posta como comentario na PR

gh pr comment 142 --body "..." — o relatorio aparece direto no GitHub.

7

📦 Resultados copiados de volta, 🗑 container destruido

Nada fica para tras. Seu projeto original nao foi tocado.

YAML do Workflow: code-review.yaml

name: code-review
version: 1
description: "Review a PR file-by-file for bugs, security, and performance"

config:
  global:
    timeout: 300s
    sandbox:
      enabled: true
      image: "node:20"
  agent:
    model: claude-sonnet-4-20250514

scopes:
  review_single_file:                     # scope reutilizavel
    steps:
      - name: review
        type: agent
        prompt: |
          Review {{ scope.value }} from this PR diff for:
          1. Bugs and logic errors
          2. Security vulnerabilities
          3. Performance issues
          4. Best practices

steps:
  - name: fetch_pr_diff
    type: cmd
    command: "gh pr diff {{ target }}"

  - name: list_changed_files
    type: cmd
    command: "gh pr diff {{ target }} --name-only"

  - name: review_files               # map paralelo!
    type: map
    scope: review_single_file
    items: "{{ steps.list_changed_files.stdout | split(pat='\n') }}"
    concurrency: 4

  - name: consolidate_report
    type: chat
    prompt: |
      Consolidate these reviews into a report ordered by severity:
      {{ steps.review_files.outputs | join(sep='\n---\n') }}

  - name: post_pr_comment
    type: cmd
    command: "gh pr comment {{ target }} --body '{{ steps.consolidate_report.response }}'"

A issue #247 descreve um bug. O minion le a issue, planeja, implementa, passa por lint gate e test gate, e abre um PR — tudo dentro do sandbox.

Pipeline completo

1

🐳 Sandbox criado

Container Docker sobe com a imagem configurada. Workspace copiado.

2

gh issue view 247 — le a issue

O titulo, corpo e labels da issue sao capturados para contexto.

3

Encontra arquivos relevantes a partir do corpo da issue

Busca por nomes de arquivo, classes ou funcoes mencionados na issue.

4

AI planeja a implementacao

Step agent cria um plano detalhado com os arquivos a modificar.

5

Gate valida que o plano e substantivo

Se o plano for vazio ou generico demais, o workflow falha antes de gastar tokens implementando.

6

AI implementa o fix

Claude Code escreve o codigo dentro do container, seguindo o plano aprovado.

7

Lint Gate (3 rounds): lint → falhou → AI corrige → lint de novo

repeat com max_iterations: 3. Se lint passar, faz break. Se falhar, o minion corrige e tenta de novo.

8

Test Gate (2 rounds): test → falhou → AI corrige → test de novo

Mesma logica do lint gate, mas rodando npm test ou equivalente.

9

Cria branch, commita, pusha, abre PR

Steps cmd deterministicos: git checkout -b, git commit, git push, gh pr create.

10

📦 Resultados copiados de volta, 🗑 container destruido

O PR ja esta no GitHub. O container e descartado.

08 — Cenario 4

Refactoring Seguro

Extrair logica, renomear, reorganizar — com lint gate e test gate garantindo que nada quebrou.

🔧 Refactoring com Safety Net sandbox

O minion usa chat (barato) para planejar e agent para implementar. Gates de lint e test garantem que o refactoring nao quebrou nada.

  • 🐳 Sandbox criado
  • chat LLM planeja o refactoring (rapido e barato)
  • agent implementa as mudancas
  • Lint gate (3 rounds): se falhar, AI corrige
  • Test gate (2 rounds): se falhar, AI corrige
  • Cria branch e PR com descricao detalhada
  • 🗑 Container destruido — se algo deu errado, nenhum dano
Terminal — Refactoring
$ minion execute refactor \ -- "extract auth logic from UserController" 🐳 Sandbox created ● plan_refactor 4.2s chat 1.2k tokens (barato!) ◉ implement 38.5s agent 6.8k tokens ✓ lint_gate 12.1s repeat(1/3) ✓ test_gate 28.3s repeat(1/2) ✓ create_pr 3.2s cmd ✓ Complete in 86.3s — 8.0k tokens — $0.05 PR: https://github.com/org/repo/pull/43 🐳 Container destroyed.
Por que chat + agent?

Usar chat para planejamento e ~10x mais barato que agent. O chat nao tem acesso ao filesystem (nao precisa), e o agent so e chamado quando realmente precisa escrever codigo. Economia inteligente de tokens.

11 — Recursos Avancados

Prompt Registry & Stack Detection

O engine detecta automaticamente a linguagem/framework do projeto e injeta os comandos corretos de lint, test, build e install nos templates. Um unico workflow funciona para Rust, Python, TypeScript, React, Java, Spring, JavaScript.

Como funciona

1

Engine analisa os arquivos do projeto

Procura por Cargo.toml, package.json, tsconfig.json, pom.xml, requirements.txt, etc.

2

Match de file_markers + content_match

Primeiro verifica se os arquivos existem (file_markers), depois verifica conteudo (content_match) para diferenciar, por exemplo, Java puro de Spring Boot.

3

Resolve com fallback chain

Se react nao definir lint, herda de typescript → javascript → _default.

4

Injeta variaveis {{ stack.* }} nos templates

O workflow usa {{ stack.tools.test }} e o engine resolve para cargo test, pytest, npm test, etc.

Arquivo prompts/registry.yaml

version: 1
detection_order:
  - java-spring    # mais especifico primeiro
  - java
  - react
  - typescript
  - javascript
  - python
  - rust

stacks:
  _default:             # fallback para stacks sem config
    tools:
      test: "echo 'no test runner configured'"

  rust:
    parent: _default
    file_markers: ["Cargo.toml"]
    tools:
      lint: "cargo clippy -- -D warnings"
      test: "cargo test"
      build: "cargo build --release"
      install: "cargo fetch"

  python:
    parent: _default
    file_markers: ["pyproject.toml", "setup.py", "requirements.txt"]
    tools:
      test: "pytest"
      install: "pip install -r requirements.txt"

  java-spring:
    parent: java          # herda de java, nao de _default
    file_markers: ["pom.xml"]
    content_match:       # verifica CONTEUDO do arquivo
      "pom.xml": "spring-boot"
    tools:
      test: "mvn test -Dspring.profiles.active=test"

Uso nos Workflows

steps:
  - name: install
    type: cmd
    run: "{{ stack.tools.install }}"     # pip install / cargo fetch / npm ci

  - name: test_gate
    type: repeat
    max_iterations: 3
    scope:
      steps:
        - name: run_tests
          type: cmd
          run: "{{ stack.tools.test }} 2>&1"  # pytest / cargo test / npm test
          on_fail: continue
        - name: fix
          type: agent
          prompt: "Fix the test failures: {{ run_tests.stderr }}"
Um workflow, todos os stacks

O mesmo fix-test.yaml funciona para Rust, Python, TypeScript, Java, etc. O engine substitui {{ stack.tools.test }} pelo comando correto baseado nos arquivos do projeto. Voce nao precisa criar workflows separados por linguagem.

12 — Recursos Avancados — Typed Outputs

Typed Output Parsing

Em vez de tratar tudo como string, declare output_type em qualquer step e o engine parseia automaticamente o resultado.

workflow.yaml — output_type em acao
steps: - name: get_metrics type: cmd run: "curl -s https://api.example.com/metrics" output_type: json # json | integer | lines | boolean | text - name: count_files type: cmd run: "find src -name '*.rs' | wc -l" output_type: integer - name: check_coverage type: cmd run: "cargo tarpaulin --print-summary | grep 'coverage'" output_type: lines - name: report type: agent prompt: | # Acesse campos tipados diretamente: Metricas: {{ get_metrics.output.cpu_usage }} Total de arquivos: {{ count_files.output }} # ja e i64, nao string Linhas de coverage: {{ check_coverage.output[0] }}
5 tipos suportados: json → serde_json::Value com acesso por dot notation • integer → i64 • lines → Vec<String> com acesso por indice • boolean → true/false • text → String (default)

13 — Recursos Avancados

Async Steps — Background Execution

Qualquer step pode rodar em background com async_exec: true. O engine faz await automaticamente quando voce referencia o output.

async-workflow.yaml — 3 tarefas em paralelo
steps: # Dispara 3 steps simultaneamente: - name: lint type: cmd run: "cargo clippy 2>&1" async_exec: true - name: test type: cmd run: "cargo test 2>&1" async_exec: true - name: audit type: cmd run: "cargo audit 2>&1" async_exec: true # Quando voce referencia o output, o engine faz await automatico: - name: report type: agent prompt: | Resultados (auto-awaited): Lint: {{ lint.output }} # ← await lint aqui Test: {{ test.output }} # ← await test aqui Audit: {{ audit.output }} # ← await audit aqui # No final do workflow, qualquer step async pendente # e await'd automaticamente (nada se perde).
Como funciona: 1. async_exec: true → tokio::spawn imediato • 2. Output referenciado em template → auto-await transparente • 3. Final do workflow → await-all de tudo que ficou pendente
14 — Recursos Avancados

Script Step — Rhai Inline

Transformacoes de dados sem spawnar processos externos. O step script usa Rhai, uma linguagem de scripting nativa em Rust com sandbox de 1M operacoes.

script-workflow.yaml — transformacao inline
steps: - name: get_data type: cmd run: 'echo "alice,bob,charlie"' - name: transform type: script run: | // Leia do contexto let raw = ctx_get("get_data"); let names = raw.split(","); let result = []; for name in names { result.push(name.to_upper()); } // Escreva de volta no contexto ctx_set("uppercase_names", result); // O ultimo valor e o output do step result.len() - name: use_result type: agent prompt: | Transformou {{ transform.output }} nomes. # ctx_set tambem injeta no contexto global
API Rhai disponivel: ctx_get(key) → le qualquer valor do contexto • ctx_set(key, value) → escreve no contexto • Limit de 1M operacoes • Sem acesso a filesystem/rede (sandboxed por design)
15 — Recursos Avancados

Chat Sessions & Truncation

Steps chat agora mantêm historico de conversa entre chamadas. Configure estrategias de truncation para nao estourar a context window.

chat-session.yaml — multi-turn com truncation
scopes: interview: steps: - name: ask type: chat prompt: "{{ scope_value }}" config: model: claude-sonnet-4-20250514 session: "main_chat" # nome da sessao truncation_strategy: last # manter ultimas N msgs truncation_count: 20 # N = 20 steps: - name: q1 type: call scope: interview initial_value: "O que e o projeto?" - name: q2 type: call scope: interview initial_value: "Quais sao os riscos?" # q2 tem acesso automatico ao historico de q1 - name: q3 type: call scope: interview initial_value: "Resuma tudo." # q3 tem q1 + q2 no historico (truncado a 20 msgs)
5 estrategias de truncation: none → manter tudo • last(N) → ultimas N mensagens • first(N) → primeiras N • first_last(F,L) → primeiras F + ultimas L • sliding_window(T) → caber em T tokens
16 — Recursos Avancados

Templates: Step Type + Accessors (?, !, from)

O template e um dos 10 step types do engine. Entenda por que ele existe e como os accessors ?, !, from() funcionam.

Por que type: template existe?

O problema que template resolve

Quando um prompt de AI fica grande (50+ linhas com instrucoes, contexto de multiplos steps, condicionais), colocar tudo inline no YAML vira um pesadelo de manutencao. E quando voce quer reutilizar o mesmo prompt em workflows diferentes, tem que copiar e colar.

O step template resolve isso: ele renderiza um arquivo .md.tera externo com zero custo de API — roda 100% localmente. O resultado pode ser usado como input para um step agent ou chat posterior.

❌ Sem template (prompt inline)

- name: implement
  type: agent
  prompt: |
    Implemente o plano abaixo.
    ## Plano
    {{ steps.plan.response }}
    ## Regras do Projeto
    {% if steps.rules is defined %}
    {{ steps.rules.stdout }}
    {% endif %}
    ## Instrucoes
    - Nao rode testes
    - Nao crie arquivos extras
    - Siga convencoes do projeto
    ... (50+ linhas de YAML)

✅ Com template (arquivo separado)

- name: build_prompt
  type: template  # zero custo
  # renderiza prompts/build_prompt.md.tera

- name: implement
  type: agent
  prompt: "{{ steps.build_prompt.response }}"

O arquivo prompts/build_prompt.md.tera contem as 50+ linhas com logica Tera, reutilizavel por qualquer workflow.

Quando usar type: template?
• Prompts complexos com 20+ linhas de instrucoes e condicionais Tera
• Mesmo prompt reutilizado em multiplos workflows (DRY)
• Gerar texto localmente (relatorios, configs, changelogs) sem chamar AI
• Compor um mega-prompt a partir de outputs de varios steps anteriores

Accessors: 3 modos de acesso a dados

Dentro de qualquer template (inline ou arquivo .md.tera), voce tem tres modos de acessar variaveis:

template-patterns.yaml — 3 accessors + from()
steps: - name: maybe_skip type: gate condition: "{{ should_analyze }}" - name: analyze type: agent prompt: "Analise os dados" - name: report type: agent prompt: | # Normal: falha se nao existir Nome do projeto: {{ project_name }} # Safe (?): retorna "" se step foi skipped Analise: {{ analyze.output? }} # Strict (!): ERRO explicito se nao existir Config obrigatoria: {{ database_url! }} # from(): acesse outputs de QUALQUER scope Dado de outro scope: {{ from("global_setup").output }} Campo aninhado: {{ from("api_call").output.data.id }}

{{ x }} — Normal

Erro se x nao existe. Use para dados obrigatorios.

{{ x? }} — Safe

Retorna "" se missing. Use para steps condicionais (gate skip).

{{ x! }} — Strict

Erro imediato com mensagem clara. Use para fail-fast em configs.

17 — Recursos Avancados

Map collect/reduce

Steps map agora coletam e reduzem resultados automaticamente. Sem precisar de um step extra para agregar.

collect-reduce.yaml — analise de modulos
scopes: analyze_module: steps: - name: review type: agent prompt: "Analise {{ scope_value }} e de nota 0-100" output_type: integer steps: - name: review_all type: map items: "auth,payments,users,notifications" scope: analyze_module parallel: 4 config: collect: json # text | all | json reduce: sum # concat | sum | count | min | max | filter # Resultado: soma das notas de todos os modulos - name: verdict type: agent prompt: | Score total: {{ review_all.output }} # e.g. 340 (soma)
Operacoes disponiveis:
collect: text (concatena stdout) • json (array JSON) • all (struct completa)
reduce: concat • sum • count • min • max • filter (com reduce_condition)
18 — Recursos Avancados

Plugin System & Event Bus

Estenda o engine com step types customizados via plugins .dylib/.so. Observe tudo com o event bus e webhooks.

Plugin System

config: plugins: - name: kubernetes path: ./plugins/k8s.dylib - name: terraform path: ./plugins/tf.so steps: - name: deploy type: kubernetes # tipo customizado! config: namespace: production image: myapp:latest

Implemente o trait PluginStep em Rust, compile como .dylib, e o engine carrega automaticamente via libloading.

Event Bus

config: events: webhook: https://slack.com/api/hook file: ./logs/events.jsonl # Eventos emitidos automaticamente: # StepStarted, StepCompleted, StepFailed # WorkflowStarted, WorkflowCompleted # SandboxCreated, SandboxDestroyed

Cada evento tem step_name, duration_ms, timestamp. Subscribers: webhook (HTTP POST) ou file (JSONL append).

19 — Guia Pratico

Setup no Seu Projeto

Como adicionar Minion Engine a qualquer projeto existente em 4 passos.

1

Instale e configure (unica vez)

# Instalar
cargo install minion-engine

# API key (obrigatorio)
echo 'export ANTHROPIC_API_KEY="sk-ant-..."' >> ~/.zshrc

# GitHub auth (o engine auto-detecta o token)
gh auth login
2

Crie um workflow a partir de template

minion init code-review --template code-review
3

Customize lint/test para seu stack (opcional com Prompt Registry)

Se seu projeto tiver prompts/registry.yaml, o engine detecta o stack automaticamente e usa os comandos corretos de lint/test via {{ stack.tools.test }}. Caso contrario, customize manualmente:

Node.js / TypeScript

lint_cmd: "npm run lint"
test_cmd: "npm test"

Python

lint_cmd: "ruff check ."
test_cmd: "pytest"

Rust

lint_cmd: "cargo clippy"
test_cmd: "cargo test"

Java / Spring

lint_cmd: "./gradlew check"
test_cmd: "./gradlew test"
4

Execute!

Sandbox ON por padrao. Docker image auto-construida na 1a vez. GH_TOKEN auto-detectado. Basta rodar:

minion execute code-review --verbose -- 142
# Docker sandbox ON por padrao — seu projeto esta seguro
Tudo automatico

Na primeira execucao o engine: (1) constroi a imagem Docker se necessario, (2) detecta o stack do projeto via prompts/registry.yaml, (3) auto-detecta GH_TOKEN do gh auth token, (4) configura credenciais de git dentro do container. Voce so precisa ter ANTHROPIC_API_KEY + Docker + gh auth login.

20 — Comparacao

🐳 Com e Sem Sandbox

O sandbox e o padrao — veja por que isso e importante.

Sem Sandbox 🐳 Com Sandbox
Seguranca ❌ Agent acessa tudo no filesystem ✔ Isolado em container Docker
Se der errado 😱 Cleanup manual, git reset, rezar 🗑 Container destruido, zero impacto
Network 🌐 Acesso total a internet 🔒 Allow/deny list configuravel
Resources ♾ Sem limite de CPU/memoria ⚙ CPU e memoria limitados
Reprodutibilidade ⚠ Depende do estado da maquina ✔ Ambiente limpo e consistente
CI/CD ⚠ Precisa de permissoes especiais ✔ Roda em qualquer runner com Docker
Padrao Seguro

O sandbox e ON por padrao — voce nao precisa fazer nada para ativa-lo. O overhead e minimo (3-5s para criar/destruir o container) e a seguranca e enorme. Use --no-sandbox apenas durante desenvolvimento local rapido de um workflow.

Flags do execute

🐳 --no-sandbox

Desativa o Docker sandbox (que e ON por padrao). Use apenas para desenvolvimento local rapido. Sem sandbox, o agent roda direto na sua maquina.

--dry-run

Mostra quais steps seriam executados sem executar nada. Arvore visual com tipos, commands e config resolvida.

--json

Output estruturado em JSON. Suprime spinners e cores. Ideal para integracao com pipelines CI/CD.

--resume <step>

Retoma execucao a partir de um step especifico. Carrega outputs anteriores do state file salvo.

--verbose

Mostra output completo de cada step, incluindo stdout/stderr de commands e tokens usados por agents.

--var KEY=VAL

Passa variaveis extras para o workflow. Acessiveis via {{ args.KEY }} nos templates Tera. Exemplo: --var mode=unit → {{ args.mode }}.

23 — FAQ

Perguntas Frequentes

Preciso de Docker instalado?
Sim, Docker Desktop 4.40+ e necessario pois o sandbox e ON por padrao. Na primeira execucao, a imagem minion-sandbox:latest e construida automaticamente — o Dockerfile esta embutido no binario, voce nao precisa fazer nada. Sem Docker, use --no-sandbox — os workflows rodam direto na sua maquina, so nao tem o isolamento do container.
Funciona com qualquer linguagem?
Sim! O Minion Engine e agnostico de linguagem. Voce so precisa customizar os comandos de lint e test no YAML do workflow. Node, Python, Rust, Go, Java, Ruby, PHP — qualquer linguagem que rode no terminal funciona. A imagem Docker do sandbox tambem e configuravel.
E seguro usar --dangerously-skip-permissions?
Dentro do sandbox, sim. O container e isolado — mesmo que o agent faca algo inesperado, nao afeta sua maquina. A flag so e perigosa quando usada sem sandbox, pois da ao Claude Code acesso irrestrito ao seu filesystem.
Posso usar em CI/CD?
Sim! Use a flag --json para output estruturado sem spinners e cores. O sandbox ja e ON por padrao, entao o isolamento no runner de CI vem de graca. Funciona com GitHub Actions, GitLab CI, CircleCI, ou qualquer plataforma que tenha Docker disponivel.
Quanto custa?
O Minion Engine em si e gratuito e open-source. Voce so paga pelo uso das APIs de AI (Anthropic/OpenAI). Um fix-issue tipico consome ~10-15k tokens (~$0.05-0.10). Steps cmd, gate, repeat, template sao gratuitos — rodam localmente sem API calls.
E se o agent fizer besteira?
Com o sandbox (padrao): o container e destruido e seu codigo original permanece intocado. Se voce usou --no-sandbox: voce pode usar --resume para retomar de um ponto anterior, ou git reset. Por isso o sandbox e o padrao — a seguranca e o principal diferencial.
Qual a diferenca entre agent e chat?
agent usa o Claude Code CLI completo — pode ler/escrever arquivos, rodar comandos no filesystem. chat faz uma chamada direta a API do LLM — mais rapido e barato, mas sem acesso ao filesystem. Use agent para implementar codigo. Use chat para planejamento, analise e sumarizacao.
Como fazer debug de um workflow?
Use --dry-run para visualizar sem executar. Use --verbose para ver output completo de cada step. Use minion inspect para ver config resolvida e dependency graph. Use --json para output parseavel em scripts.
Preciso configurar GH_TOKEN manualmente?
Nao. Se voce autenticou o gh CLI com gh auth login, o engine auto-detecta o token rodando gh auth token e injeta no container Docker automaticamente. Voce so precisa exportar GH_TOKEN manualmente se nao tiver o gh CLI instalado.
O que e Prompt Registry / Stack Detection?
O arquivo prompts/registry.yaml define os stacks do seu projeto (Rust, Python, TypeScript, React, Java, etc.) com file_markers e content_match. O engine detecta o stack automaticamente e injeta variaveis como {{ stack.tools.test }}, {{ stack.tools.lint }} nos templates. Assim um unico workflow funciona para qualquer linguagem.
O que preciso configurar apenas UMA vez?
Tres coisas: (1) cargo install minion-engine, (2) export ANTHROPIC_API_KEY="sk-ant-..." no ~/.zshrc, (3) gh auth login. A imagem Docker e construida automaticamente na primeira execucao. Depois disso, basta minion execute workflow.yaml -- target.
