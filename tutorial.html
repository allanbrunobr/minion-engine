AI workflow engine que torna seu codigo mais robusto, seguro e inteligente — tudo dentro de Docker sandboxes.

01 — Por que Minion Engine?

Seu Time de AI Rodando Localmente

Como o Stripe Minions, mas rodando na sua maquina com Docker sandboxes em vez de AWS. Cada acao roda isolada em um container — seu codigo esta seguro.

🐳

Sandbox-First

Cada step roda isolado em Docker. Se algo der errado, o container e destruido. Seu codigo original permanece intocado.

🤖

AI-Powered

Claude Code planeja, implementa, revisa e corrige — autonomamente. Voce define o workflow, os minions executam.

🔄

Self-Healing

Lint falhou? Testes quebraram? Os minions auto-corrigem e tentam de novo. Gates com retry automatico ate 3 rounds.

02 — Conceito

O que sao Minions?

Em vez de um unico agente de AI com acesso total ao seu sistema, voce tem multiplos minions especializados que trabalham em sandboxes isolados. Se um minion cometer um erro, o container e destruido — impacto zero.

Filosofia: Engine controla, Minions executam
Engine: "Minion, implemente o plano" Minion: *implementa dentro do container* Engine: "Obrigado. Agora EU rodo lint." ← deterministico Engine: "Lint falhou. Minion, corrija." Minion: *corrige dentro do container* Engine: "Lint passou. EU rodo testes." Engine: "Tudo OK. Copiando resultados de volta." Engine: "Container destruido. Seu projeto esta seguro."

Por que isso importa?

AI agents tradicionais tem acesso direto ao seu filesystem. Se o agente deletar algo errado, instalar um pacote malicioso, ou corromper sua build — voce limpa a bagunca manualmente. Com sandbox, o pior que acontece e voce destruir o container e tentar de novo.

03 — Docker Sandbox

🐳 Como Funciona o Sandbox

Tres modos de sandboxing. Escolha o nivel de isolamento ideal para seu caso.

🐳

Modo 1: --sandbox

FullWorkflow — tudo roda dentro do Docker. O workspace inteiro e copiado para o container. Resultados copiados de volta ao final.

minion execute workflow.yaml \
  --sandbox
🤖

Modo 2: AgentOnly

config.agent.sandbox: true — somente steps de AI (agent) rodam isolados. Commands rodam localmente.

config:
  agent:
    sandbox: true
⚙

Modo 3: Devbox

Full config — imagem customizada, regras de rede, limites de recursos. Controle total.

sandbox:
  enabled: true
  image: "node:20"

Configuracao Completa do Devbox

config:
  global:
    sandbox:
      enabled: true
      image: "node:20"                           # imagem Docker base
      network:
        allow: ["api.anthropic.com", "registry.npmjs.org"]  # whitelist
        deny:  ["0.0.0.0/0"]                       # bloqueia tudo mais
      resources:
        cpus: 2.0                                # limite de CPU
        memory: "4g"                              # limite de memoria
🐳 Network allow/deny

O minion so pode acessar URLs na whitelist. Isso impede que o agent faca requests inesperados, instale pacotes maliciosos de registries nao-autorizados, ou exfiltre dados. Seguranca por default.

05 — Cenario 1

Entrei num projeto — quero analisar uma PR

Voce acabou de entrar no time e quer revisar a PR #142. O minion analisa arquivo por arquivo, em paralelo, dentro de um sandbox.

Terminal — ~/projects/my-saas
$ cd ~/projects/my-saas $ minion execute workflows/code-review.yaml -- 142 --sandbox --verbose 🐳 Sandbox created (node:20, 2 CPUs, 4GB RAM) workspace copied to /workspace ✓ fetch_pr_diff 1.4s cmd gh pr diff 142 ✓ list_changed_files 0.3s cmd ◉ review_files 28.7s map(4) 8 files, concurrency:4 ✓ src/api/auth.ts 6.2s agent 2 critical ✓ src/api/users.ts 5.8s agent 1 medium ✓ src/lib/db.ts 7.1s agent clean ✓ src/lib/cache.ts 4.3s agent 1 low ✓ ... +4 files ● consolidate_report 8.3s chat Critical → High → Medium → Low ✓ post_pr_comment 1.1s cmd gh pr comment 142 ✓ Complete in 40.1s — 12.3k tokens — $0.06 🐳 Results copied back. Container destroyed.

O que aconteceu passo a passo

1

🐳 Container Docker criado, workspace copiado

Todo o diretorio do projeto e copiado para /workspace dentro do container.

2

gh pr diff 142 → busca o diff da PR

Roda dentro do container. O diff completo e salvo no Context Store.

3

Lista arquivos alterados

Extrai a lista de arquivos do diff para iterar depois.

4

Para CADA arquivo (paralelo, concurrency:4), AI revisa

Bugs, seguranca, performance, boas praticas. Cada review roda em paralelo com ate 4 minions simultaneos.

5

Consolida em relatorio (Critical → High → Medium → Low)

Um step chat (barato e rapido) sintetiza todas as reviews em um relatorio ordenado por severidade.

6

Posta como comentario na PR

gh pr comment 142 --body "..." — o relatorio aparece direto no GitHub.

7

📦 Resultados copiados de volta, 🗑 container destruido

Nada fica para tras. Seu projeto original nao foi tocado.

YAML do Workflow: code-review.yaml

name: code-review
version: 1
description: "Review a PR file-by-file for bugs, security, and performance"

config:
  global:
    timeout: 300s
    sandbox:
      enabled: true
      image: "node:20"
  agent:
    model: claude-sonnet-4-20250514

scopes:
  review_single_file:                     # scope reutilizavel
    steps:
      - name: review
        type: agent
        prompt: |
          Review {{ scope.value }} from this PR diff for:
          1. Bugs and logic errors
          2. Security vulnerabilities
          3. Performance issues
          4. Best practices

steps:
  - name: fetch_pr_diff
    type: cmd
    command: "gh pr diff {{ target }}"

  - name: list_changed_files
    type: cmd
    command: "gh pr diff {{ target }} --name-only"

  - name: review_files               # map paralelo!
    type: map
    scope: review_single_file
    items: "{{ steps.list_changed_files.stdout | split(pat='\n') }}"
    concurrency: 4

  - name: consolidate_report
    type: chat
    prompt: |
      Consolidate these reviews into a report ordered by severity:
      {{ steps.review_files.outputs | join(sep='\n---\n') }}

  - name: post_pr_comment
    type: cmd
    command: "gh pr comment {{ target }} --body '{{ steps.consolidate_report.response }}'"
06 — Cenario 2

Preciso resolver uma issue do GitHub

A issue #247 descreve um bug. O minion le a issue, planeja, implementa, passa por lint gate e test gate, e abre um PR — tudo dentro do sandbox.

Terminal — Fix Issue #247
$ minion execute workflows/fix-issue.yaml -- 247 --sandbox --verbose 🐳 Sandbox created (node:20, 2 CPUs, 4GB RAM) ✓ fetch_issue 1.2s cmd gh issue view 247 ✓ find_relevant_files 0.8s cmd ◉ plan 12.3s agent 2.1k tokens ✓ validate_plan 0.0s gate plan is substantive ◉ implement 45.7s agent 8.4k tokens ↻ lint_gate 18.2s repeat(2/3) failed round 1, fixed round 2 ✓ test_gate 34.1s repeat(1/2) passed first try ✓ create_branch 0.2s cmd ✓ commit_and_push 2.4s cmd ✓ create_pr 1.8s cmd ✓ Complete in 117.2s — 10.5k tokens — $0.08 PR: https://github.com/org/repo/pull/42 🐳 Results copied back. Container destroyed.

Pipeline completo

1

🐳 Sandbox criado

Container Docker sobe com a imagem configurada. Workspace copiado.

2

gh issue view 247 — le a issue

O titulo, corpo e labels da issue sao capturados para contexto.

3

Encontra arquivos relevantes a partir do corpo da issue

Busca por nomes de arquivo, classes ou funcoes mencionados na issue.

4

AI planeja a implementacao

Step agent cria um plano detalhado com os arquivos a modificar.

5

Gate valida que o plano e substantivo

Se o plano for vazio ou generico demais, o workflow falha antes de gastar tokens implementando.

6

AI implementa o fix

Claude Code escreve o codigo dentro do container, seguindo o plano aprovado.

7

Lint Gate (3 rounds): lint → falhou → AI corrige → lint de novo

repeat com max_iterations: 3. Se lint passar, faz break. Se falhar, o minion corrige e tenta de novo.

8

Test Gate (2 rounds): test → falhou → AI corrige → test de novo

Mesma logica do lint gate, mas rodando npm test ou equivalente.

9

Cria branch, commita, pusha, abre PR

Steps cmd deterministicos: git checkout -b, git commit, git push, gh pr create.

10

📦 Resultados copiados de volta, 🗑 container destruido

O PR ja esta no GitHub. O container e descartado.

08 — Cenario 4

Refactoring Seguro

Extrair logica, renomear, reorganizar — com lint gate e test gate garantindo que nada quebrou.

🔧 Refactoring com Safety Net sandbox

O minion usa chat (barato) para planejar e agent para implementar. Gates de lint e test garantem que o refactoring nao quebrou nada.

  • 🐳 Sandbox criado
  • chat LLM planeja o refactoring (rapido e barato)
  • agent implementa as mudancas
  • Lint gate (3 rounds): se falhar, AI corrige
  • Test gate (2 rounds): se falhar, AI corrige
  • Cria branch e PR com descricao detalhada
  • 🗑 Container destruido — se algo deu errado, nenhum dano
Terminal — Refactoring
$ minion execute workflows/refactor.yaml \ -- "extract auth logic from UserController" --sandbox 🐳 Sandbox created ● plan_refactor 4.2s chat 1.2k tokens (barato!) ◉ implement 38.5s agent 6.8k tokens ✓ lint_gate 12.1s repeat(1/3) ✓ test_gate 28.3s repeat(1/2) ✓ create_pr 3.2s cmd ✓ Complete in 86.3s — 8.0k tokens — $0.05 PR: https://github.com/org/repo/pull/43 🐳 Container destroyed.
Por que chat + agent?

Usar chat para planejamento e ~10x mais barato que agent. O chat nao tem acesso ao filesystem (nao precisa), e o agent so e chamado quando realmente precisa escrever codigo. Economia inteligente de tokens.

🎯 Flaky Test Diagnosis & Fix auto-fix

11 — Guia Pratico

Setup no Seu Projeto

Como adicionar Minion Engine a qualquer projeto existente em 5 passos.

1

Instale o minion

cargo install minion-engine  # ou brew/binary
2

Crie um workflow a partir de template

minion init code-review --template code-review
3

Customize lint/test para seu stack

Edite o YAML gerado para usar os comandos corretos do seu projeto.

Node.js

lint_cmd: "npm run lint"
test_cmd: "npm test"

Python

lint_cmd: "ruff check ."
test_cmd: "pytest"

Rust

lint_cmd: "cargo clippy"
test_cmd: "cargo test"

Go

lint_cmd: "golangci-lint run"
test_cmd: "go test ./..."
4

Ative o sandbox para seguranca

Adicione no YAML do workflow:

config:
  global:
    sandbox:
      enabled: true
5

Execute!

minion execute code-review.yaml -- 142 --sandbox --verbose
12 — Comparacao

🐳 Com e Sem Sandbox

Por que voce deveria sempre usar --sandbox.

Sem Sandbox 🐳 Com Sandbox
Seguranca ❌ Agent acessa tudo no filesystem ✔ Isolado em container Docker
Se der errado 😱 Cleanup manual, git reset, rezar 🗑 Container destruido, zero impacto
Network 🌐 Acesso total a internet 🔒 Allow/deny list configuravel
Resources ♾ Sem limite de CPU/memoria ⚙ CPU e memoria limitados
Reprodutibilidade ⚠ Depende do estado da maquina ✔ Ambiente limpo e consistente
CI/CD ⚠ Precisa de permissoes especiais ✔ Roda em qualquer runner com Docker
Recomendacao

Sempre use --sandbox em producao e CI/CD. O overhead e minimo (3-5s para criar/destruir o container) e a seguranca que voce ganha e enorme. O unico caso onde sem sandbox faz sentido e durante desenvolvimento local rapido de um workflow.

13 — CLI Reference

Todos os Comandos

Referencia rapida do minion CLI.

ComandoDescricaoFlags Principais
minion execute <yaml> -- [target] Executa um workflow --verbose --quiet --json --dry-run --sandbox --resume <step> --var KEY=VAL --args '{}'
minion validate <yaml> Valida YAML sem executar —
minion list Lista workflows disponiveis —
minion init <name> Cria workflow a partir de template --template <name> --output <dir>
minion inspect <yaml> Mostra config resolvida e dependency graph —

Flags do execute

🐳 --sandbox

Roda o workflow inteiro em Docker sandbox isolado. Requer Docker Desktop 4.40+. O workspace e copiado para o container e resultados copiados de volta.

--dry-run

Mostra quais steps seriam executados sem executar nada. Arvore visual com tipos, commands e config resolvida.

--json

Output estruturado em JSON. Suprime spinners e cores. Ideal para integracao com pipelines CI/CD.

--resume <step>

Retoma execucao a partir de um step especifico. Carrega outputs anteriores do state file salvo.

--verbose

Mostra output completo de cada step, incluindo stdout/stderr de commands e tokens usados por agents.

--var KEY=VAL

Passa variaveis extras para o workflow. Acessiveis via {{ vars.KEY }} nos templates Tera.

15 — FAQ

Perguntas Frequentes

Preciso de Docker instalado?
Sim, Docker Desktop 4.40+ e necessario para usar o modo sandbox (--sandbox). Sem Docker, voce ainda pode usar o Minion Engine normalmente — os workflows rodam direto na sua maquina, so nao tem o isolamento do container.
Funciona com qualquer linguagem?
Sim! O Minion Engine e agnostico de linguagem. Voce so precisa customizar os comandos de lint e test no YAML do workflow. Node, Python, Rust, Go, Java, Ruby, PHP — qualquer linguagem que rode no terminal funciona. A imagem Docker do sandbox tambem e configuravel.
E seguro usar --dangerously-skip-permissions?
Dentro do sandbox, sim. O container e isolado — mesmo que o agent faca algo inesperado, nao afeta sua maquina. A flag so e perigosa quando usada sem sandbox, pois da ao Claude Code acesso irrestrito ao seu filesystem.
Posso usar em CI/CD?
Sim! Use a flag --json para output estruturado sem spinners e cores. Combine com --sandbox para isolamento no runner de CI. Funciona com GitHub Actions, GitLab CI, CircleCI, ou qualquer plataforma que tenha Docker disponivel.
Quanto custa?
O Minion Engine em si e gratuito e open-source. Voce so paga pelo uso das APIs de AI (Anthropic/OpenAI). Um fix-issue tipico consome ~10-15k tokens (~$0.05-0.10). Steps cmd, gate, repeat, template sao gratuitos — rodam localmente sem API calls.
E se o agent fizer besteira?
Com sandbox: o container e destruido e seu codigo original permanece intocado. Sem sandbox: voce pode usar --resume para retomar de um ponto anterior, ou git reset. Mas e por isso que recomendamos sempre usar --sandbox — a seguranca e o principal diferencial.
Qual a diferenca entre agent e chat?
agent usa o Claude Code CLI completo — pode ler/escrever arquivos, rodar comandos no filesystem. chat faz uma chamada direta a API do LLM — mais rapido e barato, mas sem acesso ao filesystem. Use agent para implementar codigo. Use chat para planejamento, analise e sumarizacao.
Como fazer debug de um workflow?
Use --dry-run para visualizar sem executar. Use --verbose para ver output completo de cada step. Use minion inspect para ver config resolvida e dependency graph. Use --json para output parseavel em scripts.
